# AI 日历 (AI Calendar) 产品需求文档

## 1. 产品概述

AI 日历是一款以大语言模型 (LLM) 为核心驱动的智能日程管理工具。产品通过自然语言交互（文本 / 语音）替代传统日历中繁琐的表单录入，让用户能够像聊天一样完成日程规划与生活记录。

与传统日历不同，AI 日历不仅支持创建未来待办和提醒，也支持碎片化记录已经发生的事项。系统通过 Agent 自动识别时间语义、事件时态与主题分类，帮助用户沉淀“过去的轨迹”和“未来的计划”，形成统一的时间管理与记忆辅助系统。

## 2. 产品目标

### 2.1 愿景

构建一个能够理解自然语言、持续记录个人时间轨迹，并主动协助规划未来安排的智能时间助理。

### 2.2 目标用户

- 研发人员、研究人员、学生等需要记录工作流和学习轨迹的人群
- 习惯使用碎片时间进行语音记录的高频日程用户
- 对模型能力、隐私和 API 灵活性有要求的开发者用户

### 2.3 核心价值

- 降低日程记录门槛，减少手动填表操作
- 将“过去记录”和“未来规划”统一到同一个时间系统中
- 通过 BYOK 支持用户自定义模型能力和数据控制方式

## 3. 核心使用场景

### 3.1 研发与学术记录

用户可以直接输入或说出类似“下午两点到四点在处理数据库查询优化和实验”的描述，系统自动识别时间区间、事件内容与类别，并将其保存为已发生的工作轨迹，便于日后复盘和检索。

### 3.2 碎片化日程安排

用户随时发送类似“下周二早上 9 点提醒我开组会”的语句，系统自动解析相对时间、生成未来事件并写入日历视图。

### 3.3 自定义 AI 大脑 (BYOK)

用户可以接入 OpenAI、Gemini、Anthropic 或自定义兼容接口，让产品既适合普通用户，也适合有隐私、成本或模型偏好要求的进阶用户。

## 4. 功能需求

### 4.1 多模态交互模块

#### 4.1.1 输入方式

- 支持文本输入
- 支持一键语音输入
- 聊天式录入为核心交互方式

#### 4.1.2 ASR 转换

- 语音内容需尽可能高精度地转写为文本
- 转写结果作为 Agent 解析输入
- 需保留原始转写文本以便用户校对和追溯

### 4.2 Agent 智能解析引擎

这是产品的核心能力模块。

#### 4.2.1 时态识别

- 准确区分“已发生事件”和“待发生事件”
- 支持过去、现在、未来三类时间语义的判断
- 支持混合表达，例如“我昨天整理了资料，明天下午继续开会”

#### 4.2.2 时间戳转换

- 将“昨天傍晚”“今晚”“下周三”“两小时后”等自然语言时间表达解析为绝对时间
- 解析应结合用户时区、当前时间和上下文
- 支持开始时间、结束时间、持续时长三类信息推断

#### 4.2.3 自动分类与高亮

- 根据事件语义自动打标签
- 为不同类别配置颜色编码
- 初始建议分类规则如下：

| 类别 | 示例 | 颜色建议 |
| --- | --- | --- |
| 学术 / 研发 | 实验、写代码、论文、调研 | 红色 |
| 生活 | 吃饭、运动、购物、通勤 | 绿色 |
| 会议 | 组会、面试、讨论、客户沟通 | 蓝色 |

#### 4.2.4 信息补全交互

- 当解析结果缺少必要字段时，Agent 需主动追问
- 若仅给出开始时间，可引导补充结束时间或采用默认时长
- 系统应支持合理默认值，例如默认 30 分钟或 1 小时

#### 4.2.5 结构化输出

Agent 需要将自然语言转化为可存储的结构化对象，建议至少包含以下字段：

- `title`
- `description`
- `start_at`
- `end_at`
- `tense` (`past` / `future`)
- `category`
- `color`
- `source_type` (`text` / `voice`)
- `raw_input`
- `confidence`

### 4.3 双轨视图展示模块

#### 4.3.1 时间轴视图 (Timeline)

时间轴应围绕“当前时间”组织，分为过去区与未来区。

- 过去区：显示已完成轨迹，位于当前时间线上方，使用灰色遮罩或弱化样式
- 未来区：显示计划行程，位于当前时间线下方，保留颜色标签和提醒状态
- 当前时间节点应始终清晰可见

#### 4.3.2 对话流视图 (Chat)

- 展示用户与 Agent 的历史交互
- 每条对话与解析生成的时间事件保持关联
- 对话更新后，时间轴实时同步

#### 4.3.3 联动体验

- 点击时间轴事件，可回看对应输入上下文
- 点击聊天记录，可高亮对应事件卡片
- 保持“说什么”与“生成了什么”之间的可追踪性

### 4.4 BYOK 模型配置中心

#### 4.4.1 提供商选择

- 支持 OpenAI
- 支持 Gemini
- 支持 Anthropic
- 预留更多供应商扩展能力

#### 4.4.2 高级自定义

- 支持自定义 Base URL
- 支持兼容 OpenAI 协议的中转服务
- 支持本地部署模型，例如 Ollama

#### 4.4.3 凭证管理

- API Key 必须加密存储
- 不在前端明文展示完整密钥
- 支持更新、删除和状态检查

#### 4.4.4 连通性测试

- 提供一键 Test 功能
- 返回模型是否可达、响应延迟和基础响应结果
- 明确区分“认证失败”“网络失败”“模型不可用”等异常类型

## 5. 非功能需求

### 5.1 性能

- 普通文本事件解析目标响应时间小于 3 秒
- 模型连通性测试应返回明确耗时指标
- 时间轴更新应尽量接近实时

### 5.2 安全

- API Key 采用加密存储
- 用户时间数据传输需使用 HTTPS
- 后端需限制日志中泄露敏感输入与凭证

### 5.3 可扩展性

- 模型供应商接入方式应统一抽象
- 事件结构支持后续增加提醒、地点、参与人等字段
- 支持未来扩展多端同步与团队协作场景

## 6. 技术架构建议

### 6.1 前端展示层

候选方案：

- Flutter
- 微信小程序
- Web：React + FullCalendar + Tailwind CSS

建议优先级：

- 如果目标是尽快验证 MVP，优先 Web
- 如果目标是移动端体验优先，可后续扩展 Flutter 或小程序

### 6.2 后端逻辑层

- Python + FastAPI
- 使用 LiteLLM 统一封装多模型供应商接口
- 将语音转写、Agent 解析、事件落库和配置中心接口分别模块化

### 6.3 数据存储层

- PostgreSQL 作为主数据库
- 适合处理日历类时序数据
- 具备较强查询和一致性能力

建议初始数据实体包括：

- `users`
- `events`
- `chat_messages`
- `model_configs`
- `voice_transcripts`

### 6.4 部署环境

- 后端服务、数据库及密钥管理模块统一部署在阿里云服务器
- 优先保障中国大陆网络访问连通性
- 为模型请求、数据库和加密模块设置隔离策略与备份机制

## 7. UI/UX 设计规范摘要

### 7.1 视觉风格

- 极简主义布局
- 大面积留白
- 悬浮对话框采用毛玻璃效果 (Glassmorphism)

### 7.2 动效规范

- 对话气泡使用打字机效果
- 语音输入时展示 AI 主色波纹动画
- 日历事件块在解析完成后以平滑“生长”动画出现

### 7.3 品牌与配色建议

- AI 主色可使用 `#7B61FF`
- 历史轨迹区域使用中性灰弱化视觉重量
- 分类色需兼顾可读性与时间轴层级感

## 8. MVP 范围建议

建议第一阶段只覆盖最核心闭环：

- 文本输入创建过去 / 未来事件
- 基础时间语义解析
- 时间轴视图展示
- 聊天记录展示
- OpenAI 兼容模型的 BYOK 配置
- PostgreSQL 持久化存储

暂不纳入 MVP 的内容：

- 多端同步
- 团队协作
- 复杂提醒规则
- 高级统计分析
- 多语种深度适配

## 9. 成功标准

MVP 可用的最低判断标准：

- 用户可通过一句自然语言成功创建事件
- 系统可识别该事件属于过去记录或未来计划
- 时间轴与聊天视图能正确联动展示
- 用户可完成自定义模型配置并通过测试接口验证连接

## 10. 后续执行建议

可直接基于本 PRD 拆分以下工作流：

1. 信息架构和交互稿设计
2. 数据表设计与 API 设计
3. Agent 提示词与结构化输出协议设计
4. MVP 前后端开发
5. 模型接入与密钥安全策略实现
6. 小范围用户测试与迭代
